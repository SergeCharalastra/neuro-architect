# Интеграция нейро-сметчика через NeuroAgents

## 1. Цель интеграции

Создать ИИ-сотрудника в `app.neuroagents.pro`, который принимает первичные заявки на расчет смет по строительству и монтажу экстерьера:

- террасы
- веранды
- навесы
- крыльцо
- пристройки
- смежные наружные работы

Главная цель: не заменить инженера-сметчика полностью, а автоматизировать первый слой работы:

1. Быстро ответить клиенту.
2. Понять тип объекта.
3. Собрать вводные.
4. Определить, можно ли дать предварительный расчет.
5. Сформировать структурированную карточку заявки.
6. Передать расчет в таблицу, менеджеру или CRM.
7. Вернуть клиенту аккуратный предварительный ответ без ложной точности.

## 2. Главное решение

NeuroAgents используем как фронт общения с клиентом.

Расчет, прайсы, нормы и контроль качества выносим отдельно:

```text
Клиент
-> NeuroAgents
-> сбор вводных и квалификация
-> переменные заявки
-> Google Sheets / n8n / HTTP endpoint
-> расчет и проверка
-> ответ клиенту / передача менеджеру / CRM
```

Почему так:

- LLM хорошо ведет диалог, уточняет, объясняет и структурирует.
- Таблица лучше считает, хранит цены, нормы, коэффициенты и формулы.
- Менеджер нужен для спорных конструкций, финальной цены и коммерческого решения.

Критическое правило:

> Агент не имеет права придумывать цены, объемы и конструктивные решения. Если данных или цены нет, он сообщает, что расчет предварительный и требует проверки.

## 3. Роли в системе

### 3.1. NeuroAgents

Отвечает за:

- первичный диалог
- квалификацию клиента
- сбор вводных
- объяснение процесса
- заполнение переменных
- передачу данных в таблицу, CRM или менеджеру
- возврат предварительного ответа

Не отвечает за:

- финальную инженерную смету
- юридически значимое коммерческое предложение
- расчет без прайса
- выбор конструкции без проверки специалистом
- обещание точной цены без осмотра/проверки проекта

### 3.2. Google Sheets / Excel

Отвечает за:

- прайс материалов
- прайс работ
- расходники
- логистику
- коэффициенты
- резерв
- формулы расчета
- итоговые суммы
- версию для клиента

### 3.3. Codex

Используется как рабочая мастерская проекта:

- проектирует промпты
- готовит базу знаний
- создает схемы данных
- готовит таблицы и CSV
- пишет тестовые сценарии
- улучшает логику агента
- анализирует ошибки после тестов
- поддерживает документацию внедрения

### 3.4. Менеджер / сметчик

Проверяет:

- спорные объемы
- выбранный фундамент
- доступ к участку
- актуальность цен
- состав работ
- финальную маржу
- коммерческие условия

## 4. MVP-сценарий запуска

Первый запуск делаем без сложной автоматизации.

### MVP 1

```text
NeuroAgents
-> собирает данные
-> формирует карточку заявки
-> менеджер переносит/проверяет данные в таблице
-> клиент получает предварительный расчет
```

Что проверяем:

- бот задает правильные вопросы
- бот не выдумывает цену
- бот понимает, когда данных недостаточно
- менеджеру удобно работать с карточкой
- клиент не теряется в диалоге

### MVP 2

```text
NeuroAgents
-> собирает данные
-> отправляет данные через HTTP/n8n
-> расчетный модуль/таблица считает
-> NeuroAgents возвращает клиенту предварительный диапазон
```

Запускать только после проверки MVP 1.

## 5. Основные режимы агента

### Режим 1. Смета по ручным вводным

Когда использовать:

- клиент знает примерные размеры
- проекта нет
- нужно быстро оценить порядок бюджета

Что агент собирает:

- тип объекта
- длина
- ширина
- высота, если важна
- площадь
- материал настила
- наличие кровли
- наличие остекления
- фундамент
- регион/локация
- доступ к участку
- срочность
- пожелания по бюджету

Результат:

- предварительная структура сметы
- список недостающих данных
- рекомендация передать менеджеру для уточнения

### Режим 2. Смета по готовому проекту

Когда использовать:

- клиент присылает PDF, чертежи, скриншоты, ведомости

Что агент должен сделать:

- принять файл или запросить его
- объяснить, что проект требует проверки
- выделить, какие данные нужны из проекта
- не обещать точный расчет автоматически

Что извлекать из проекта:

- площадь террасы/веранды
- площадь кровли
- площадь остекления
- фундамент
- количество свай
- бетон
- арматура
- пиломатериал
- кровельные материалы
- утепление
- пленки/мембраны
- проемы
- ведомости и спецификации

Результат:

- протокол извлечения
- список найденных данных
- список не найденных данных
- список спорных мест
- передача менеджеру или в расчетную таблицу

### Режим 3. Проект + замены

Когда использовать:

- клиент хочет заменить материал
- клиент хочет убрать или добавить работы
- клиент выбирает другой фундамент
- клиент хочет несколько вариантов

Правило:

> Альтернативные варианты нельзя складывать в одну смету. Их нужно считать отдельными версиями.

Пример:

- Вариант А: сваи + ростверк
- Вариант Б: плита
- Вариант В: существующее основание

### Режим 4. Проверка чужой сметы

Когда использовать:

- клиент прислал смету от другого подрядчика
- хочет понять, что завышено или пропущено

Что проверять:

- пропущенные работы
- завышенные позиции
- непонятные формулировки
- отсутствующие объемы
- отсутствие логистики
- отсутствие расходников
- отсутствие резерва
- конфликт с проектом

Результат:

- список рисков
- вопросы к подрядчику
- рекомендации по уточнению
- предложение сделать альтернативный расчет

## 8. База знаний для NeuroAgents

Базу знаний лучше делить на короткие блоки.

### Блок 1. Роль агента

Агент является первичным сметчиком-консультантом. Он помогает клиенту понять, какие данные нужны для расчета, собирает вводные и передает их в расчет или менеджеру.

### Блок 2. Что агент не делает

Агент не:

- выдумывает цены
- обещает финальную стоимость
- выбирает конструкцию без проверки специалистом
- скрывает неопределенность
- складывает альтернативные варианты
- выдает предварительную оценку как точную смету

### Блок 3. Обязательные вопросы

Для ручного расчета:

1. Что нужно построить или рассчитать?
2. Какие размеры объекта?
3. Есть ли кровля?
4. Есть ли остекление?
5. Какой материал настила/каркаса хотите?
6. Какой фундамент планируется?
7. Где находится объект?
8. Есть ли проект, фото, чертеж или пример?
9. Нужна предварительная оценка или точная смета?

### Блок 4. Стоп-правила

Агент обязан остановить точный расчет и передать на проверку, если:

- нет размеров
- не выбран фундамент
- проект содержит несколько вариантов конструкции
- цена в прайсе отсутствует
- клиент просит финальную цену без проверки
- есть демонтаж, сложный доступ или нестандартные работы
- данные клиента противоречат проекту

### Блок 5. Клиентское объяснение

Базовая формулировка:

```text
Могу собрать предварительную структуру сметы и показать порядок бюджета. Для точной стоимости нужно подтвердить материалы, фундамент, доступ к участку и актуальные цены. Если хотите, я соберу данные и передам специалисту для финальной проверки.
```

### Что хранить в нормах

```text

```

### Ключевые правила расчета

- Настил считается по площади.
- Кровля считается по площади кровли.
- Остекление считается по площади остекления.
- Сваи считаются по штукам.
- Доставка считается по рейсу или километражу.
- Сложность применяется к работам.
- Резерв применяется к предварительному итогу.
- Альтернативные фундаменты считаются отдельными версиями.

## 10. Интеграционные варианты

### Вариант A. Ручной MVP

NeuroAgents собирает карточку.

Менеджер вручную переносит данные в таблицу.

Подходит для:

- первого теста
- проверки сценария
- быстрой настройки
- минимального риска

Минус:

- часть работы остается ручной

### Вариант B. Google Sheets через n8n

NeuroAgents отправляет данные в n8n.

n8n записывает строку в Google Sheets.

Таблица считает.

n8n возвращает результат или уведомляет менеджера.

Подходит для:

- устойчивого MVP
- первых клиентов
- понятного контроля

Минус:

- нужно настроить n8n-сценарий

### Вариант C. HTTP endpoint

NeuroAgents отправляет данные на API.

API валидирует данные, считает смету, возвращает ответ.

Подходит для:

- масштабирования
- более сложных расчетов
- интеграции с CRM

Минус:

- нужна разработка endpoint

### Рекомендация

Начать с варианта A, затем перейти к варианту B.

Вариант C имеет смысл только после 20-30 реальных заявок, когда будет понятно, какие поля и ошибки повторяются.

## 11. Логика ответа клиенту

### Если данных достаточно для предварительной оценки

Ответ:

```text
По вашим вводным можно собрать предварительную смету. Я вижу такие основные блоки: основание, каркас, настил, кровля/остекление, работы и логистика. Сейчас расчет будет предварительным, потому что финальную стоимость нужно сверить по материалам, доступу к участку и актуальному прайсу.
```

### Если данных недостаточно

Ответ:

```text
Пока данных недостаточно для корректной сметы. Чтобы не назвать случайную сумму, мне нужно уточнить несколько параметров: размеры, тип основания, материал настила и наличие кровли. После этого я смогу собрать структуру расчета и передать ее специалисту.
```

### Если есть проект

Ответ:

```text
Проект можно использовать как основу расчета. Сначала нужно извлечь из него площади, спецификации, фундамент, кровлю, остекление и спорные места. После этого можно собрать предварительную смету и отдельно отметить, что требует проверки специалистом.
```

### Если клиент просит точную цену сразу

Ответ:

```text
Точную цену без проверки проекта, материалов и условий участка лучше не называть: есть риск ошибиться по объему или составу работ. Я могу сейчас собрать предварительную оценку и список уточнений, а финальную стоимость подтвердит специалист.
```

## 12. Передача менеджеру

Менеджеру передавать заявку, если:

- клиент оставил телефон
- клиент прислал проект
- есть готовность считать
- есть спорный фундамент
- есть нестандартные работы
- клиент просит точную цену
- в расчетах не хватает ключевых данных

Формат комментария менеджеру:

```text
Клиент хочет расчет: [тип объекта].
Известно: [размеры, материалы, локация].
Не хватает: [список].
Риски: [список].
Рекомендуемый следующий шаг: [позвонить / запросить проект / сделать предварительный расчет / отправить КП].
```

## 13. Тестовые сценарии перед запуском

Перед реальным трафиком проверить минимум 10 сценариев:

1. Клиент знает размеры и материал.
2. Клиент не знает площадь.
3. Клиент прислал проект.
4. Клиент просит точную цену сразу.
5. Клиент выбирает между сваями и плитой.
6. Клиент хочет заменить материал.
7. Клиент прислал чужую смету.
8. Клиент не оставляет телефон.
9. Клиент пишет коротко: "сколько стоит веранда?"
10. Клиент описывает сложный участок.

Критерий успеха:

- агент не выдумывает цену
- задает не больше 3-5 вопросов подряд
- собирает обязательные поля
- помечает недостающие данные
- корректно передает менеджеру
- не обещает финальную стоимость без проверки

## 14. Риски

### Риск 1. Ложная точность

Бот может выглядеть уверенно и назвать сумму без достаточных данных.

Защита:

- стоп-правила
- таблица как источник цены
- обязательная пометка "предварительно"

### Риск 2. Слишком длинный диалог

Клиент может устать от вопросов.

Защита:

- максимум 3-5 вопросов за один шаг
- сначала собрать минимум
- затем предложить передать менеджеру

### Риск 3. PDF-разбор слабее ожиданий

Проекты могут быть сложными, сканы могут плохо читаться.

Защита:

- не обещать автоматический точный разбор
- делать протокол найденных данных
- спорные позиции отдавать менеджеру

### Риск 4. Прайс устаревает

Цены материалов быстро меняются.

Защита:

- дата проверки цены
- источник цены
- поле активности позиции
- запрет на точную цену при пустом/старом прайсе

## 16. Критерий успеха MVP

MVP считается рабочим, если:

- 80% заявок получают корректную первичную квалификацию
- менеджер получает структурированную карточку
- клиент понимает следующий шаг
- бот не выдумывает финальную цену
- расчет можно воспроизвести в таблице
- время первичной обработки заявки снижается минимум на 50%
