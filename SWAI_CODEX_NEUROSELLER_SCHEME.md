# Анализ схемы "Нейропродавец для свай" и адаптация под Codex

## 1. Главный вывод

Схема из чата сильная: она правильно переводит идею "GPT-продавца" из абстрактного помощника в прикладной инструмент продаж для узкой услуги.

Лучшее в ней:
- фокус на одной нише: винтовые сваи;
- цель не "отвечать", а вести к микро-конверсии;
- разделение инструкции и базы знаний;
- работа со стадиями лида;
- запрет на выдумывание цены, сроков и технических обещаний;
- обязательный тест на реальных сценариях;
- правильный вывод по Telegram: не GPTs напрямую, а Telegram Bot + n8n/Make/backend + OpenAI API.

Главная слабость: схема собрана под Custom GPTs, где модель живет внутри ChatGPT. Для Codex ее нужно переосмыслить не как "один GPT", а как проектную систему файлов, сценариев, тестов, артефактов и автоматизаций.

## 2. Что я вижу в чате

Чат фактически строит MVP нейропродавца для услуги закручивания винтовых свай.

Внутри есть 5 слоев:

1. Позиционирование агента:
   - нейропродавец-консультант по свайным основаниям;
   - не универсальный помощник;
   - не инженер-проектировщик и не сметчик без данных;
   - цель: вести к замеру, звонку, фото, расчету или заявке.

2. Инструкция поведения:
   - коротко;
   - живо;
   - без давления;
   - без канцелярита;
   - отвечать по стадии клиента;
   - не обещать без данных.

3. Knowledge-pack:
   - услуги и объекты;
   - частые вопросы и возражения;
   - квалификация лида;
   - цены и условия;
   - тон и примеры диалогов;
   - регионы и логистика;
   - реальные сценарии;
   - готовые ответы;
   - ошибки и антипаттерны;
   - критерии хорошего ответа и конверсии.

4. Тестирование:
   - 10-20 тестовых диалогов;
   - смотреть, где агент много спрашивает, звучит шаблонно, не дает вывода, обещает лишнее или не ведет к следующему шагу.

5. Интеграция:
   - GPTs Actions не использовать на старте;
   - для Telegram нужен отдельный бот;
   - лучший путь: Telegram Bot + n8n + OpenAI API;
   - GPTs использовать как эталон логики, а не как канал доставки.

## 3. Что работает

### Узкая специализация

Схема не пытается сделать "ИИ для продаж вообще". Она строит продавца под одну услугу: винтовые сваи под ключ.

Почему это работает:
- меньше расплывчатости;
- проще собрать базу знаний;
- проще тестировать;
- проще измерять конверсию;
- легче контролировать стиль и ошибки.

### Фокус на следующем шаге

Самая сильная идея: хороший ответ не просто "правильный", а двигает клиента дальше.

Успешные микро-конверсии:
- клиент прислал фото;
- дал размеры;
- назвал район;
- согласился на звонок;
- согласился на замер;
- уточнил задачу;
- перестал сопротивляться.

Это правильнее, чем оценивать бота по "красоте ответов".

### Разделение инструкции и knowledge

Инструкция отвечает за поведение.
Knowledge отвечает за предметную базу.

Это сильная архитектура, потому что:
- инструкцию можно держать компактной;
- знания можно обновлять отдельно;
- ошибки легче находить;
- можно добавлять реальные кейсы и переписки без переписывания ядра.

### Антигаллюцинационные правила

Очень важный блок: не называть точную цену, срок, пригодность участка и возможность проезда техники без данных.

Для строительной ниши это критично. Ошибка в ответе может создать ложное ожидание, конфликт с клиентом или финансовый риск.

### Правильная логика Telegram

Вывод в чате верный: Custom GPT нельзя просто "поставить в Telegram" как готового продавца. Для Telegram нужна отдельная архитектура:

Telegram Bot -> n8n/Make/backend -> OpenAI API -> Telegram response

Это дает:
- контроль контекста;
- логирование;
- передачу лида;
- статусы;
- уведомления менеджеру;
- подключение CRM;
- human handoff.

## 4. Что не работает или требует усиления

### 1. Много текстовой базы, но мало операционной схемы

Knowledge-pack хороший, но пока он больше похож на методичку. Для реального внедрения не хватает карты состояний лида.

Нужно явно задать:
- холодный лид;
- теплый лид;
- горячий лид;
- сложный объект;
- клиент просит цену;
- клиент прислал фото;
- клиент готов к замеру;
- клиент спорит или сомневается;
- клиент молчит после ответа.

Для каждого состояния нужен следующий шаг.

### 2. Нет четкой CRM-модели

В чате есть идея "лид в CRM", но нет минимальной структуры данных.

Нужно фиксировать:
- имя;
- контакт;
- канал;
- объект;
- размеры;
- район;
- фото есть/нет;
- барьер клиента;
- стадия лида;
- следующий шаг;
- краткое резюме диалога;
- статус передачи человеку.

Без этого бот будет разговаривать, но бизнес не будет видеть систему.

### 3. Не хватает тестового набора

Есть рекомендация "прогнать 10-20 сценариев", но сами сценарии надо оформить как отдельный test pack.

Минимум:
- клиент спрашивает цену;
- клиент сравнивает с бетоном;
- клиент говорит "дорого";
- клиент "просто узнать";
- клиент присылает фото;
- клиент просит быстрый выезд;
- сложный участок;
- удаленный район;
- горячий лид;
- конфликтный клиент.

### 4. Риск слишком мягкого продавца

Схема хорошо избегает давления, но может уйти в чрезмерную осторожность.

Риск:
- бот будет постоянно просить фото/детали;
- не будет закрывать на действие;
- горячий лид остынет.

Нужно добавить правило:
если клиент уже просит выезд, срок, контакт или готов прислать фото, не продолжать квалификацию, а переводить к действию.

### 5. Нет метрик эффективности

Для MVP нужны простые метрики:
- процент диалогов, где получено фото;
- процент диалогов, где получен контакт;
- процент переходов к замеру;
- среднее число сообщений до микро-конверсии;
- доля диалогов, переданных человеку;
- частые причины отказа.

Без метрик это будет "бот, который вроде отвечает".

## 5. Как улучшить схему

### Улучшение 1. Добавить карту состояний

Формат:

| Состояние клиента | Признак | Цель ответа | Следующий шаг |
|---|---|---|---|
| Холодный | "просто узнать" | дать ясность | 1 вопрос по задаче |
| Цена | "сколько стоит" | объяснить зависимости | фото/размеры |
| Сомнение | "подойдет ли" | снять риск | объект + фото |
| Сравнение | "бетон или сваи" | сравнить под задачу | уточнить объект |
| Горячий | "когда можно приехать" | не тормозить | район + контакт |
| Фото | клиент прислал фото | интерпретировать осторожно | звонок/замер |

### Улучшение 2. Добавить модель данных лида

Минимальная карточка:

```json
{
  "name": "",
  "contact": "",
  "channel": "telegram",
  "object_type": "",
  "object_size": "",
  "location": "",
  "has_photo": false,
  "client_barrier": "",
  "lead_stage": "cold|warm|hot",
  "next_step": "ask_photo|ask_size|call|site_visit|human_handoff",
  "summary": "",
  "created_at": ""
}
```

### Улучшение 3. Сжать knowledge для первого запуска

Не начинать с 10 файлов, если цель - быстро проверить MVP.

Лучший старт:
1. `01_Услуги_и_объекты.md`
2. `02_FAQ_и_возражения.md`
3. `03_Квалификация_и_стадии_лида.md`
4. `04_Тон_и_готовые_ответы.md`
5. `05_Ограничения_и_антипаттерны.md`

Потом добавить:
- цены;
- регионы;
- кейсы;
- реальные переписки;
- тесты.

### Улучшение 4. Ввести human handoff

Бот должен знать, когда передать человеку.

Передавать человеку, если:
- клиент готов к замеру;
- клиент оставил контакт;
- есть конфликт или недоверие;
- задача нестандартная;
- клиент прислал фото сложного участка;
- нужна точная цена;
- нужен быстрый выезд.

### Улучшение 5. Сделать отдельный файл тестов

Codex должен не только генерировать ответы, но и проверять качество по чек-листу.

Критерии:
- ответ короткий;
- есть польза;
- не выдумана цена;
- не обещано лишнее;
- стадия лида определена;
- есть следующий шаг;
- нет давления;
- горячий лид не заторможен.

## 6. Как адаптировать под Codex

### Главное отличие Codex от Custom GPTs

Custom GPTs - это "настроенный чат-бот".
Codex - это рабочая среда, где можно хранить, менять, тестировать и развивать систему как проект.

Поэтому в Codex схему лучше оформлять не как один большой промт, а как набор файлов:

```text
/neuro-seller-svai
  /knowledge
    01_services.md
    02_faq_objections.md
    03_lead_qualification.md
    04_tone_answers.md
    05_limits_antipatterns.md
    06_prices_conditions.md
    07_regions_logistics.md
    08_cases.md
  /prompts
    system_prompt.md
    response_policy.md
    handoff_policy.md
  /tests
    lead_scenarios.md
    answer_quality_checklist.md
    regression_cases.md
  /automation
    telegram_n8n_flow.md
    lead_schema.json
    crm_fields.md
```

### Как Codex должен работать с этой схемой

Codex может быть не самим продавцом, а архитектором и редактором системы:

1. Создавать и поддерживать knowledge-файлы.
2. Переписывать системный prompt.
3. Генерировать тестовые диалоги.
4. Проверять ответы по чек-листу.
5. Проектировать n8n-сценарий.
6. Описывать структуру лида для CRM.
7. Готовить тексты для сайта, квиза, рекламы и Telegram.
8. Улучшать логику после реальных переписок.

### Практический Codex-workflow

1. Собрать базовый пакет:
   - системный prompt;
   - 5 knowledge-файлов;
   - 10 тестовых сценариев;
   - lead schema.

2. Прогнать тест:
   - Codex генерирует ответы нейропродавца;
   - затем оценивает их по чек-листу;
   - затем предлагает правки в prompt/knowledge.

3. Подготовить Telegram MVP:
   - BotFather token;
   - n8n Telegram Trigger;
   - OpenAI API call;
   - memory by chat_id;
   - lead extraction;
   - Telegram response;
   - manager notification.

4. Запустить пилот:
   - 20-50 диалогов;
   - вручную смотреть ошибки;
   - обновлять knowledge.

5. Масштабировать:
   - CRM;
   - статусы лидов;
   - автосводка менеджеру;
   - аналитика возражений;
   - ретаргетинг по стадиям.

## 7. Рекомендуемая версия под Codex

Для проекта лучше сделать не "Свайный Нейропродавец GPT", а продуктовый модуль:

**AI-продавец для монтажников винтовых свай**

Состав MVP:
- лендинг;
- Telegram-бот;
- база знаний;
- сценарии ответов;
- квалификация лида;
- передача заявки менеджеру;
- таблица/CRM;
- тестовый набор;
- еженедельное улучшение по реальным диалогам.

Ценность для бизнеса:
- быстрее отвечать на входящие заявки;
- не терять лиды вечером и в выходные;
- собирать фото, размеры и район до звонка;
- снижать нагрузку на менеджера;
- повышать долю заявок, дошедших до замера;
- видеть частые возражения и улучшать сайт/оффер.

## 8. Минимальный тест

Запустить MVP не на всю воронку, а на один сценарий:

**"Клиент спрашивает стоимость монтажа свай"**

Бот должен:
1. Не назвать цену из воздуха.
2. Коротко объяснить, от чего зависит стоимость.
3. Попросить 1-2 ключевых ввода: объект, размеры или фото.
4. Определить стадию лида.
5. Если клиент теплый или горячий - предложить звонок/замер.
6. Передать лид менеджеру при готовности.

Критерий успеха:
- из 30 диалогов минимум 10 дают фото, размеры, район или контакт;
- ни в одном ответе бот не обещает точную цену без данных;
- менеджер получает понятное резюме лида.

2. `knowledge/01_services.md`
3. `knowledge/02_faq_objections.md`
4. `knowledge/03_lead_qualification.md`
